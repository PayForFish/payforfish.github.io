研究JUC–CompletableFuture

WHY：

多线程编程在业务场景下，一般都是，使用线程池进行自定义任务的分发，然后进行结果的收集。使用JUC的同步工具可以较简单实现这些需求，但面对复杂多变的需求，往往少不了手动进行异步编程的场景。比如多个线程之间有先后依赖；多个线程结果需要进行组合再进行一些操作等。

常规的多线程编程可以解决这些问题，有经验的coder还可以优雅地封装一定的通用Util。而CompletableFuture就像是大神Doug Lea给开发者封装的这样一套工具类。

细节：

公共类CompletableFuture是CompletionStage接口的主要实现，因此需要主要关注的是CompletionStage。

关于stage

stage: (n) 阶段;段;状态;步;步骤

官方把每个线程执行结束（返回或者不返回值）的那一完成态称作stage，并以此为基础规定了此接口方法实现的规范。 CompletableFuture类就遵循它的规范，比如下一个stage执行取得上一个stage执行结果处理，若取得的value是null，则抛出NPE。

实际上所有的方法都是回调函数，在单独的Future执行完成返回时使用或者不使用线程返回结果去调用自定义的函数。

关于方法模板

CompletionStage中定义的每个方法其实都有固定的重载逻辑，比如基础的thenApply()就会衍生出异步的版本thenApplyAsync()，然后异步版本会有使用ForkJoinPool(默认)以及使用传入的自定义Executor两种方式。因此，每个方法逻辑实际上一共有3种实现。

关于异步执行

默认的方法是同步执行的，即会在注册回调函数的线程上执行。若使用异步版本API，默认使用forkJoinPooll线程调度策略，或者传入自定义Executor调度，这相当于把注册的回调函数放到“线程池”中执行，API帮助开发者封装了其中线程间通信等细节，避免陷入手动异步编程的细节里。另外需要注意的是，由于ForkJoin的工作量窃取逻辑，可能会导致异步回调函数在同一个线程中执行。

API

1. thenApply

使用一个CompletionStage实例调用thenApply()，将在上一个stage结束时将返回结果作为参数，传入调用者传入的参数-Function函数-

并在此基础上提供异步版本，缺省参数下使用默认的ForkJoinPool

2. thenAccept

thenAccept函数的入参是Consumer，因此相比于thenApply函数，它只负责消费值，没有返回值。

3. thenRun

thenRun只接收一个runnable实例作为参数，因此既没有返回值，也没有上一个stage返回的入参可以使用

4. thenCombine

thenCombine传入了另一个stage，以及一个合并函数，合并函数的两个参数分别是两个stage的返回结果，并且这个合并函数需要返回一个新的stage的执行结果。
